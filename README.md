# Orbit2 — Communardo Partner Performance Platform

Fresh build, replacing earlier drafts. Tracks Communardo's performance as an Atlassian partner (extensible to AWS, Google, GSIs, and future vendors) across 8 weighted metric categories, with a live dashboard, automated news/sentiment monitoring, and on-demand report generation.

## Folder structure

```
Orbit2/
  data/
    weights.json                category weights per vendor (source of truth for scoring)
    sales_performance.csv
    marketing.csv
    market_visibility.csv
    ai_adoption.csv
    business_planning_qbr.csv
    registrations.csv
    third_party_coselling.csv   merged 3rd Party Vendors + Co-selling (AWS/Google/GSIs)
    solutions.csv               solutions created/sold/revenue/vertical coverage
    solution_verticals.csv      per-vertical breakdown (Finance, Automotive, Pharma, etc.)
    services.csv                services revenue sold, number of clients
    news_log.csv                 market visibility / sentiment mentions (auto-appended every 12h)
    evidence_index.csv           Evidence Library ledger — see below
    categories.json              registry of all scorecard categories — add a new one here, no code changes
    metric_changelog.csv         history of every metric added / amended / retired, with reason + source
    deals.csv                    key deals register — company, TCV/ACV, products/services sold, vertical, Atlassian tier
    scores_snapshot.json         computed output — do not edit by hand, regenerated by scoring.py
  evidence_library/
    inbox/                       drop new files here (Finder or dashboard drag-and-drop)
    sales_performance/, marketing/, ... one folder per category — filed evidence lives here
  scripts/
    scoring.py                  the scoring engine
    evidence_ingest.py          files evidence + applies the dedupe/supersede rule
    metric_manager.py           add/amend/retire metrics + categories, with automatic changelog
  scripts_node/
    generate_report.js          Word report generator (scores, explanations, key deals, full glossary)
    generate_pptx.js            PowerPoint deck generator
  docs/
    methodology.md              full explanation of every formula + the Evidence Library + metric management workflow
  reports/                      generated report files land here
  dashboard_rendered.html       source HTML behind the live Orbit2 dashboard artifact
```

## Keeping metrics aligned with the vendor

When Atlassian (or another vendor) renames, adds, or retires a metric, use the dashboard's **Manage Metrics** panel — edit weights/targets, add or remove sub-metric rows, or add a whole new category, then click "Submit changes to Claude" to apply them. Or drop a vendor program document into the **Import vendor program update** box next to it (distinct from the Evidence Library box — that one's for evidence behind an existing score, this one's for changes to the metrics themselves). Every change is logged to `data/metric_changelog.csv` and shown on the dashboard, so you always know what changed and why. Full detail in `docs/methodology.md`.

## Evidence Library

Every screenshot, CSV/Excel export, or document behind a metric gets filed in `evidence_library/` and logged in `data/evidence_index.csv`. Drop a file onto the dashboard's Evidence Library panel and click **Submit for analysis** (or drop it into `evidence_library/inbox/`, or attach it in chat) — processing is automatic from there: Claude checks it's actually relevant to Communardo's vendor performance (rejecting it with an explanation if not), extracts the number, updates the metric, files the evidence, and refreshes the dashboard, all without a confirmation round-trip. If a new file covers the same vendor + category + sub-metric + quarter as something already on file, the old evidence is automatically marked **superseded** and the new one becomes the active source — so the scorecard always reflects your latest evidence instead of double-counting or averaging stale and fresh data. Nothing is deleted; old evidence stays for audit history. Search past evidence (active or superseded) from the dashboard's search box or `python3 scripts/evidence_ingest.py search "<text>"`. Full detail in `docs/methodology.md`.

## How to update your data

1. Get the raw numbers — from a CSV/Excel export, or by reading a screenshot of a partner portal.
2. Open the relevant CSV in `data/` and add or edit the row for the current quarter (same columns every time: vendor, quarter, sub_metric, weight_pct_in_category, target, actual, unit, score_method, source, notes).
3. Ask Claude to re-run the scoring engine and refresh the dashboard — it will run `scripts/scoring.py` and update the Orbit2 artifact so the new numbers show up immediately.

Screenshots: drop the image into this project folder and ask Claude to read the numbers off it and update the right CSV row — no manual transcription needed on your end.

## Generating a report

The dashboard has a **Reports** tab (next to Dashboard) that renders the full report content in-page — category detail with plain-language descriptions for every sub-metric, the key deals table (company, TCV/ACV, products/services sold, vertical, Atlassian tier), and a full metric glossary — always live from the current data, no generation needed just to look at it.

To get a file, click one of the three download buttons on the Reports tab, or say "generate my Orbit2 report":
- **Download Word (.docx)** — runs `scripts_node/generate_report.js`.
- **Download PDF** — runs `generate_report.js` then converts it with LibreOffice (`scripts/office/soffice.py --headless --convert-to pdf`, from the docx skill).
- **Download PowerPoint (.pptx)** — runs `scripts_node/generate_pptx.js`, a 6-slide deck (overall score, category breakdown, key deals, market visibility & sentiment, methodology).

All three re-run `scripts/scoring.py` first so the output always reflects the current CSVs. Run any of them yourself with `cd scripts_node && npm install && node generate_report.js Atlassian` / `node generate_pptx.js Atlassian`.

## Adding a vendor

Add a weights entry in `data/weights.json` and rows tagged with the new vendor name in each CSV — see `docs/methodology.md` for the exact steps.

## GitHub + live weblink (GitHub Pages)

Orbit2 can be published as a real, public URL (e.g. `https://smojsak-code.github.io/orbit2/`) that:
- persists across sessions — the data lives in the GitHub repo, not in Claude's temporary sandbox
- always reflects the latest data — the page fetches `data/web_snapshot.json` fresh on every load, so pushing new data to GitHub is all that's needed to update it (no republishing the page itself)
- serves Word/PDF/PowerPoint reports as plain downloadable files from `reports/`

**How it's built:** `scripts/build_web.py` regenerates `data/web_snapshot.json` and the report files, and copies `web/index_template.html` to `index.html` at the repo root. `index.html` is a read-only mirror of the dashboard (view scores, download reports, search evidence) — adding evidence or changing metrics still happens in the Cowork workspace, since GitHub Pages has no backend to run that processing.

**To make it live, two things need to happen once:**
1. **Create the repo + a token** — an empty repo at `github.com/smojsak-code/orbit2`, and a Personal Access Token (Settings → Developer settings → Tokens, `repo` scope) so Claude can push on your behalf.
2. **Enable Pages** — once the repo exists and has a first push, go to its Settings → Pages → Source → "Deploy from a branch" → branch `main`, folder `/ (root)` → Save. (This step can't be done from Claude's sandbox — GitHub's settings API isn't reachable from here — so it's a one-time click on your end.)

After that, updating the live site is just: `python3 scripts/build_web.py`, then commit + push `data/`, `reports/`, and `index.html` (only changes when the template itself changes) — which Claude can do automatically going forward once the token is in place.
3. From then on, after updating data, run `git add -A && git commit -m "update data" && git push` yourself (or paste the CSV changes to Claude and ask it to walk you through the commit message).

This is the one piece of "connects to GitHub" that has to run on your machine, not in Claude's sandbox — see the honest assessment in the main chat reply for why.
